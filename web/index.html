Chunking is the #1 silent failure in RAG: if an answer is split across two chunks, retrieval can miss it. These control the split.

How this project is built

An Agentic RAG system: it answers questions about your documents and cites the exact source (document, page, score). The LLM is given the retriever as a tool and decides what to search and how many times — so multi-part questions get full coverage.

Flow

UPLOAD:  files → extract (table-aware) → chunk → embed (dense + sparse) → vector DB
ASK:     question (+ memory) → [ LLM → search tool → retrieve ]* → cited answer (streamed)
                                 │
            retrieval: dense + sparse → RRF fusion → cross-encoder rerank → top-k

Where is the data stored?

Vectors live in the vector store you choose on Settings: an in-memory Qdrant (inside the server process — zero setup, but cleared on restart) or your own Qdrant Cloud cluster (persistent). Your uploaded files are saved on the server only long enough to ingest them; the searchable data is the vectors in the store.
